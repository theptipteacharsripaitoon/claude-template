# Database & Migrations

Extends `CLAUDE.md` (especially §19 Reliability & Resource Safety). When this skill loads, its rules and Done criteria apply on top of the universal baseline.

## The cardinal rule: zero-downtime via expand → migrate → contract

**Never make a backwards-incompatible schema change in a single deploy.** Split into phases that ship independently. The application and database must always be compatible during the rollout window.

```
Phase 1: EXPAND     Add new schema, keep old. Code writes both.
Phase 2: MIGRATE    Backfill data; switch reads to new.
Phase 3: CONTRACT   Stop writing old; drop old. (Later release.)
```

Concrete: renaming a column.
1. **Expand:** add `new_column` (nullable, no constraints). Deploy code that writes to BOTH `old_column` and `new_column`.
2. **Backfill:** run a backfill job to copy `old_column` → `new_column` for existing rows.
3. **Migrate reads:** deploy code that reads from `new_column`.
4. **Stop writing old:** deploy code that writes only `new_column`.
5. **Contract:** in a later release, drop `old_column` after confirming nothing reads it.

**Never drop a column or table in the same release that stops using it.** A rollback would lose data.

## Reversibility

- **Reversible by default.** Provide both `up` and `down` (or whatever your tool's terminology is) unless a safe `down` is genuinely impossible.
- If `down` is unsafe (e.g., dropping a column with new data), document why and provide a manual recovery procedure instead of an executable destructive `down`. Do not silently omit reversibility.
- When a `down` exists, test it in CI: `up`, then `down`, then `up` again — the schema must be identical. Migrations with a documented-unsafe `down` are exempt from the round-trip.

## Lock-safe DDL

Different databases lock differently. Know your engine's gotchas. Below: PostgreSQL.

**Safe (typically fast, no rewrite):**
- `ALTER TABLE ... ADD COLUMN col TYPE NULL` (no default, no NOT NULL).
- `CREATE INDEX CONCURRENTLY` (Postgres).
- `ALTER TABLE ... ADD CONSTRAINT ... NOT VALID` then `VALIDATE CONSTRAINT` separately.
- `DROP INDEX CONCURRENTLY`.

**Dangerous (full table rewrite or long lock):**
- `ALTER TABLE ... ADD COLUMN ... NOT NULL DEFAULT <expr>` on a large table (Postgres <11; ≥11 is fast for constants).
- `ALTER TABLE ... ALTER COLUMN ... TYPE` — rewrites table.
- `ALTER TABLE ... ADD FOREIGN KEY` without `NOT VALID` — locks both tables.
- `CREATE INDEX` (without `CONCURRENTLY`) — blocks writes.
- `VACUUM FULL` — exclusive lock.

**Pattern for adding NOT NULL to existing column:**
1. Add column nullable.
2. Backfill values in batches.
3. Add `CHECK (col IS NOT NULL) NOT VALID`.
4. `VALIDATE CONSTRAINT` (online).
5. `ALTER COLUMN ... SET NOT NULL` (now fast — Postgres uses the validated check).

**Pattern for adding a foreign key without locking writes:**
```sql
ALTER TABLE orders ADD CONSTRAINT fk_user FOREIGN KEY (user_id) REFERENCES users(id) NOT VALID;
ALTER TABLE orders VALIDATE CONSTRAINT fk_user;  -- separate transaction, online
```

## Index hygiene

- **Index before the query needs it in production**, not after a slow-query alert.
- Use `CREATE INDEX CONCURRENTLY` in Postgres. Plain `CREATE INDEX` blocks writes.
- Drop unused indexes. They cost write performance and storage. Audit with `pg_stat_user_indexes`.
- Cover the most selective columns first in composite indexes.
- Partial indexes for hot subsets (`WHERE status = 'active'`).
- Don't index every column "just in case" — each index slows writes.

## Foreign keys, constraints, and cascades

- **All foreign keys explicit and named.** Anonymous constraints are unmanageable.
- **No silent `ON DELETE CASCADE`** on user data. Cascades have deleted entire user accounts in production. Prefer `RESTRICT` and explicit cleanup logic.
- **Use `ON DELETE SET NULL`** when the relationship is optional.
- **Validate constraints separately from creation** for large tables (`NOT VALID` then `VALIDATE`).

## Defaults and nullability

- New columns: default to nullable. Adding NOT NULL after backfill is the safe path.
- For Postgres ≥11: `ALTER TABLE ADD COLUMN col INT NOT NULL DEFAULT 0` is fast (stored in catalog, no rewrite).
- For older versions: split into nullable add → backfill → NOT NULL.

## Timestamps and identifiers

- **Primary keys:** prefer `BIGSERIAL` / `BIGINT GENERATED BY DEFAULT AS IDENTITY` over `INTEGER`. `INT` runs out at 2.1B and is a 12-month outage waiting to happen.
- **UUIDs:** if needed, use UUIDv7 (or ULID) for time-ordered, index-friendly IDs. Random UUIDv4 fragments index pages.
- **Timestamps:** always `TIMESTAMP WITH TIME ZONE` (Postgres) / `TIMESTAMPTZ`. Never naive timestamps.
- **`created_at`, `updated_at`** on every mutable table. Set via DB trigger or app — be consistent.
- **Soft deletes** (`deleted_at`) only when business requires; otherwise just delete. Soft deletes complicate every query.

## Naming

- **Tables:** plural, snake_case (`users`, `order_items`). One project, one convention.
- **Columns:** snake_case. Booleans named `is_*` / `has_*`.
- **Foreign keys:** `<referenced_table_singular>_id` (`user_id`, `order_id`).
- **Indexes:** `idx_<table>_<columns>` or your migration tool's convention.
- **Constraints:** `<table>_<column>_<type>` (`users_email_unique`, `orders_status_check`).

## Idempotency and safety

- Migrations should be idempotent where possible (`IF NOT EXISTS`, `ON CONFLICT DO NOTHING` for seed inserts).
- **Wrap DDL in transactions** where the engine allows. Note: Postgres `CREATE INDEX CONCURRENTLY` cannot be in a transaction.
- **Test migrations against a production-sized snapshot** before running on production. Migrations that take 10ms on dev can take hours on production.
- **Run migrations in a controlled deploy step**, not as part of app startup. App-startup migrations cause race conditions and split-brain.

## Backfills

- **Batch large backfills.** A single `UPDATE big_table SET ...` locks rows for the whole run.
- Batch size: 1k–10k rows at a time, with brief pauses to allow other work.
- Track progress (a temp table or watermark column) so the job is resumable.
- Consider running outside the migration tool entirely for very large data.

## Forbidden in migrations

- `DROP TABLE` or `DROP COLUMN` in the same release the code stops using it.
- DML for non-trivial data (>10k rows) inside migration files — use a separate, observable, restartable backfill job.
- Application-specific logic (calling external services, complex business rules). Migrations are about structure; logic belongs in app code.
- `TRUNCATE` on tables with data, period. Make it `DELETE FROM ... WHERE ...` so you can review and roll back.
- Editing existing migration files after they've run on any environment. Add a new migration instead.

## Tooling checklist

- Use one migration tool consistently: Alembic (Python), Prisma Migrate, Drizzle Kit, Flyway, golang-migrate, Sqitch. Pick one.
- Migrations live in version control, in a single ordered chain (timestamp or sequence prefix).
- CI runs `up` then `down` then `up` to verify reversibility.
- CI runs migrations against a snapshot of production schema (or close to it).
- Lint migrations: `squawk` (Postgres) catches dangerous DDL patterns automatically.

## Done criteria (in addition to CLAUDE.md §14)

- [ ] Migration is reversible (`up` + `down`), or `down` unsafety is documented.
- [ ] No backwards-incompatible change in a single deploy (expand-migrate-contract used).
- [ ] No long-locking DDL on large tables (used `CONCURRENTLY`, `NOT VALID + VALIDATE`, or split phases).
- [ ] Indexes added with `CONCURRENTLY` (Postgres).
- [ ] Foreign keys named and reviewed for cascade behavior.
- [ ] Backfill (if any) is batched, resumable, and runs outside the migration tool for large data.
- [ ] Migration tested against production-sized data; expected duration documented in PR.
- [ ] `squawk` (or equivalent) lint passes.
- [ ] Migration does not run on app startup; runs as a controlled deploy step.
